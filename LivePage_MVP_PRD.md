## 1. 背景与问题

传统远程演示依赖 PPT + 视频会议：内容静态、无法运行真实代码或原型，视频流量大、弱网体验差，且客户端安装成本高。技术评审、代码走查、原型演示等场景真正需要的是**可交互的内容**与**低负担的实时讨论**。

LivePage 用**交互式 HTML 替代 PPT**、用**纯语音通话替代视频**，做一个纯浏览器、可自部署的轻量开源会议系统。

## 2. 目标与非目标

**目标**
- 打开浏览器即可创建/加入会议，无需安装客户端。
- 多人纯语音实时通话，弱网可用。
- 主持人演示 HTML 页面，参会者操作实时同步（滚动/点击）。
- 单机 Docker Compose 一键部署，2 核 4G 可运行。

**非目标（MVP 不做）**
- 视频/屏幕共享流。
- 会议录制与回放。
- 账号体系、组织管理、计费。
- 移动端原生 App。
- 白板、聊天室、文件网盘等附加协作功能。

## 3. 目标用户与典型场景

| 用户 | 场景 | 核心诉求 |
|------|------|----------|
| 研发团队 | 代码评审 | 加载高亮代码页，语音讨论并同步滚动 |
| 技术分享者 | 技术分享 | 用 reveal.js 演示，现场跑代码示例 |
| 设计师 | 设计走查 | 加载交互原型 HTML，边看边聊 |
| 讲师 | 远程培训 | 操作演示页，学员跟随观看 |

## 4. 用户故事

- 作为主持人，我能创建带密码的会议，并拿到可携带密码的分享链接发给他人。
- 作为参会者，我能凭房间号 + 密码（或分享链接）加入会议并听到所有人语音。
- 作为主持人，我能加载一个 HTML 页面并让所有人同步看到我的操作。
- 作为参会者，我能脱离主持人视角自行浏览，再一键回到主持人视角。
- 作为主持人，我能静音、踢出成员，或把主持权移交他人。
- 作为主持人，我掉线后重连，会议仍然存在。

## 5. 功能需求

### 5.1 创建/加入会议
- 主持人创建会议，系统生成唯一房间号，并设置房间密码。
- 生成的分享链接可携带密码（如 `/room/{id}?k={token}`），点击即免手动输入。
- 参会者通过“房间号 + 密码”或分享链接加入；无需注册。密码校验在服务端完成，链接中的 token 一次性/有时效，避免明文长期暴露。
- 加入时填写昵称，允许重名（用会话 ID 区分）。

### 5.2 多人纯语音通话
- 基于 mediasoup SFU，支持多人并发上下行音频。
- 加入时申请麦克风权限；支持本地静音/取消静音。
- 参会者列表显示每人的说话/静音状态指示。

### 5.3 HTML 页面加载与同步演示
- **加载方式**：主持人上传演示内容，由服务端托管为同源内容后在 iframe 中渲染。
- **上传规格**：
  - 支持多文件上传（`.html` 主文件 + CSS/JS/图片等关联资源），保留相对目录结构。
  - 支持打包上传 `.zip`，服务端解压后托管，自动识别入口 `index.html`（或让主持人指定入口文件）。
  - 单次上传总体积上限 50MB；超限或非白名单类型时拒绝并提示。
- 主持人的滚动、点击等操作通过事件广播同步给所有跟随者。
- 同步以事件（滚动位置、点击目标定位）为粒度，低带宽传输。
- 内容为同源托管，可直接注入监听脚本，实现完整双向同步与事件回放。
- 事件同步使用稳定选择器（如元素路径/`data-*` 锚点）而非绝对坐标，减小分辨率差异带来的错位。

### 5.4 跟随/脱离浏览
- 参会者默认“跟随”主持人视角。
- 参会者可“脱离”自行浏览当前页面，界面提示已脱离。
- 提供“一键回归”，立即同步回主持人当前状态。

### 5.5 主持人权限控制
- 静音指定成员、踢出成员。
- 移交主持权给其他成员。
- 将脱离的成员“拉回”到主持人视角。

### 5.6 参会者列表与状态
- 实时展示成员列表、角色（主持人/参会者）、语音与跟随状态。

### 5.7 会议保留与重连
- 主持人离线后会议不解散，状态保留。
- 主持人重连后恢复主持权与当前演示状态。

### 5.8 部署
- Docker Compose 一键启动前端、信令、SFU、Nginx。

## 6. 非功能需求

- **性能**：单房间人数上限 100 人；同步事件端到端延迟目标 < 300ms。语音随人数增长按需限制并发上行（如仅主持人+发言者上行），保障弱网体验。
- **容量评估**：不设固定并发房间数目标，以“10 人”为资源评估单元。给出单个 10 人单元的 CPU/内存/带宽基准，总承载量按单元线性折算，据此规划部署与横向扩容（房间数与人数分布不影响单元口径）。
- **兼容**：支持最新版 Chrome/Edge/Firefox 桌面端；需 HTTPS 以启用 WebRTC 与麦克风。
- **弱网**：音频优先保障，同步事件在断连后自动重连补齐。
- **资源**：默认部署目标 2 核 4G；以 10 人单元基准折算可承载的总人数/房间数。
- **安全**：房间号不可枚举；加入需密码校验，分享链接 token 有时效；信令与媒体走 TLS；不持久化音频。上传内容做内容隔离（沙箱 iframe、限制脚本能力）、体积（≤50MB）与类型白名单校验，zip 解压需防路径穿越（zip slip）。

## 7. 技术架构

| 层 | 技术 | 职责 |
|----|------|------|
| 前端 | React 18 + Vite + Ant Design | 会议 UI（基于 antd 组件）、iframe 演示容器、事件采集 |
| 信令 | Node.js + Socket.io | 房间管理、状态广播、权限指令 |
| 语音 | mediasoup（WebRTC SFU） | 多人音频转发 |
| 网关 | Nginx | 反向代理与 TLS 终止 |
| 部署 | Docker Compose | 服务编排 |

关键流程：加入（房间号 + 密码/链接 token 校验）→ Socket.io 建立信令连接 → mediasoup 协商音频传输 → 订阅演示事件通道 → 加载演示内容（本地上传，服务端同源托管）→ iframe 渲染并跟随。

## 8. 关键指标（成功标准）

- 从打开链接到听到语音 < 15s。
- 演示同步操作可感知延迟 < 300ms。
- 主持人重连后 5s 内恢复会议状态。

## 9. 里程碑

1. M1：房间创建/加入 + 纯语音通话跑通。
2. M2：HTML 加载 + 操作同步（跟随/脱离/回归）。
3. M3：主持人权限控制 + 断线重连。
4. M4：Docker Compose 一键部署 + 文档完善。

## 10. 风险与对策

| 风险 | 对策 |
|------|------|
| 上传 HTML 含恶意脚本 | 沙箱 iframe、CSP 限制、体积与类型校验 |
| 100 人房间语音带宽压力 | 限制并发上行、按发言者动态订阅、可配置 TURN |
| mediasoup 部署与 NAT 穿透复杂 | 提供默认 TURN 配置与部署文档 |
| 弱网下同步事件堆积 | 事件去抖 + 断连后仅补最新状态 |

## 11. 待确认问题（Open Questions）

- 分享链接携带的密码 token 有效期与是否一次性？
- 100 人场景下语音上行策略：全员可发言，还是仅主持人授权者可发言？
