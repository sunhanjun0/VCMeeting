## 0. 相较 v1 的核心变更

v1 把 LivePage 定位为「带内置语音的完整会议系统」，重型 mediasoup SFU 稀释了产品真正的差异化价值。v2 重新聚焦：

| 维度 | v1 | v2 | 理由 |
|------|----|----|------|
| 产品定位 | 开源会议系统 | **实时演示同步工具** | 唯一无人做好的是「跨设备高保真操作同步」，集中火力于此 |
| 语音架构 | mediasoup SFU | **纯 WebRTC P2P Mesh** | 复用同步引擎已有的 Socket.io 信令，后端零增量成本 |
| NAT 穿透 | 必需 TURN，复杂 ICE | 公共 STUN + 可选 coturn | 部署从「专家级」降到「一条命令」 |
| 后端 | 信令 + SFU | 纯 Socket.io | 单进程，1C2G 绰绰有余 |
| 权限 | 静音/踢人/移交 | **静音/移交/拉回**（不做踢人） | 精简权限模型，保留移交 |
| 单房语音 | 100 | **≤10 自由发言，10~20 强制 PTT** | Mesh 物理上限约 20 人；>20 走 v1.1 的 LiveKit |
| 内容加载 | 上传（多文件/zip） | **上传（多文件/zip）** | 保持；同源托管是脚本注入的前提 |

大场景（20~100 人）不在 MVP 范围，未来通过可替换的 `VoiceProvider` 接口接入 LiveKit（见 §7.4）。

## 1. 背景与问题

传统远程演示依赖 PPT + 视频会议：内容静态、无法运行真实代码或原型；技术评审、代码走查、原型演示真正需要的是**可交互的内容**与**低负担的实时讨论**。LivePage 用**交互式 HTML 替代 PPT**、用**轻量 P2P 语音**完成小团队远程演示同步，纯浏览器、可自部署。

## 2. 目标与非目标

**目标**
- 打开浏览器即可建/加入房间，无需安装客户端。
- 主持人上传并演示交互式 HTML，跟随者的视角实时同步（滚动/点击/页内导航）。
- 内置纯语音（P2P Mesh）：≤10 人自由发言，10~20 人强制按键发音（PTT）。
- 单机 Docker Compose 一键部署，1C2G 可运行。

**非目标（MVP 不做）**
- 视频/屏幕共享、会议录制回放。
- 真·大规模房间（>20 人）与 SFU（走 v1.1 的 LiveKit）。
- 踢出成员。
- 账号体系、组织管理、计费。
- 白板、聊天室、文件网盘。

## 3. 目标用户与典型场景

| 用户 | 场景 | 核心诉求 | 典型人数 |
|------|------|----------|---------|
| 研发团队 | 代码评审 | 加载高亮代码页，语音讨论并同步滚动 | 3~8 |
| 设计师 | 设计走查 | 加载交互原型 HTML，边看边聊 | 3~10 |
| 技术分享者 | 小型分享 | 用 reveal.js 演示，现场跑示例 | 5~10 |

## 4. 用户故事

- 作为主持人，我能建带密码的房间，拿到可携带 token 的分享链接。
- 作为参会者，我凭房间号+密码（或分享链接）加入并听到所有人语音。
- 作为主持人，我上传一份 HTML 内容并让所有人同步看到我的滚动/点击。
- 作为参会者，我能脱离主持人视角自行浏览，再一键回归。
- 作为主持人，我能静音成员、把脱离者拉回。
- 作为主持人，我掉线重连后房间仍在、演示状态恢复。

## 5. 功能需求

### 5.1 创建/加入房间
- 主持人创建房间，系统生成不可枚举的房间号并设置密码。
- 分享链接可携带有时效 token（如 `/room/{id}?k={token}`），点击免手动输入；密码校验在服务端完成。
- **token 有效期：默认 24 小时**（可配置），且随房间销毁立即失效；**非一次性**（同一链接需供多名参会者使用）。过期后回退到「房间号 + 密码」手动加入。
- 参会者填写昵称加入，允许重名（用会话 ID 区分）。

### 5.2 演示内容加载（上传，同源托管）
- 主持人上传演示内容，服务端托管为**同源**内容后在 iframe 渲染（同源是注入同步脚本的前提）。
- 支持多文件上传（`.html` 主文件 + CSS/JS/图片，保留相对目录结构）与 `.zip` 打包上传（解压后自动识别入口 `index.html`，或主持人指定入口）。
- 单次上传总体积 ≤50MB；超限或非白名单类型拒绝并提示。
- 安全：sandbox iframe（`allow-scripts allow-same-origin`）+ CSP 限制脚本能力；zip 解压防路径穿越（zip slip）；外链/表单提交做白名单拦截。

### 5.3 操作同步引擎（核心）
向主持人与跟随者的 iframe 注入同一脚本：主持人采集事件上报，跟随者接收并应用。

- **滚动同步——视口锚点法**：主持人上报「当前视口顶部元素 + 进入该元素的偏移比例」，跟随者将同一元素滚到顶并对齐比例。抗分辨率/字号差异，避免绝对像素错位。
- **点击/交互同步——稳定选择器**：选择器优先级 `id` → `data-sync-id` → 基于 `nth-of-type` 的稳定 CSS 路径；跟随者按选择器派发合成点击（支撑 reveal.js 翻页、手风琴展开等）。
- **页内导航同步**：主持人切换 bundle 内页面/hash 路由时广播新地址，跟随者加载。
- **事件去抖**：滚动事件节流；断连后仅补最新状态，不回放积压。
- **延迟目标**：端到端同步 < 150ms。

### 5.4 跟随/脱离浏览
- 跟随者默认跟随主持人视角。
- 手动滚动/交互 → 自动脱离，顶部横幅提示「已脱离主持人视角」。
- 「回到主持人视角」按钮一键重新对齐当前状态。

### 5.5 内置语音（P2P Mesh + 按键发音）
- 基于原生 WebRTC，音频 P2P 全网状互连；SDP/ICE 交换复用 §5.1 的 Socket.io 信令通道。
- NAT 穿透：默认公共 STUN；可选自部署 coturn 兜底对称 NAT 用户。
- **发言模式随人数自适应**：
  - **≤10 人**：自由发言（开麦即说），支持本地静音/取消静音。
  - **10~20 人**：**强制按键发音（PTT）**——仅在按住发言键时上行音频，压低同时上行的带宽。
  - **>20 人**：超出 Mesh 物理上限（连接数与发言者编码次数无法靠 PTT 缓解），MVP 拒绝并提示；真·不限人数走 v1.1 的 `LiveKitProvider`（见 §7.4）。
- 主持人可静音指定成员。
- 参会者列表显示每人的说话/静音状态。

> **设计备注**：PTT 只降低「带宽」，无法降低 Mesh 的「每客户端 N-1 条连接」与「发言者编码 N-1 份」两项开销，故 Mesh 仍有约 20 人的硬顶。真正的「不限人数 + 少数发言」是 SFU 的能力，留给 v1.1。

### 5.6 主持人权限控制
- 静音指定成员。
- 将脱离的成员「拉回」主持人视角。
- **移交主持权**给指定成员：接收方获得演示/权限控制与 `latestState` 写入权，原主持人降为普通参会者；移交指令经服务端校验后广播，全员 UI 同步更新角色。
- （踢出成员：MVP 不做。）

### 5.7 参会者列表与状态
- 实时展示成员列表、角色（主持人/参会者）、语音与跟随状态。

### 5.8 会议保留与重连
- 房间只维护 `latestState`（视口锚点 + 当前页 + 时间戳）。
- 新人加入直接拉取快照即时同步。
- 主持人掉线状态冻结、房间不解散；凭 token 重连恢复主持权与演示状态（目标 5s 内）。
- **一个状态模型同时解决「首次同步 / 断线补齐 / 主持人重连」三件事。**

### 5.9 部署
- Docker Compose 一键启动前端、Socket.io 后端、可选 coturn、Nginx。

## 6. 非功能需求

- **性能**：同步事件端到端 < 150ms；语音 ≤10 人自由发言，10~20 人 PTT（同时上行受按键约束），Mesh 硬上限约 20 人。
- **兼容**：最新版 Chrome/Edge/Firefox 桌面端；需 HTTPS 以启用 WebRTC 与麦克风。
- **弱网**：音频优先保障；同步事件断连后自动重连补齐最新状态。
- **资源**：默认部署目标 1C2G（无 SFU 后瓶颈消失）。
- **安全**：房间号不可枚举；加入需密码校验，分享链接 token 有时效；信令与媒体走 TLS；不持久化音频；上传内容做沙箱隔离、体积（≤50MB）与类型白名单校验，zip 解压防 zip slip。

## 7. 技术架构

| 层 | 技术 | 职责 |
|----|------|------|
| 前端 | React 18 + Vite + Ant Design | 会议 UI、iframe 演示容器、事件采集/应用、WebRTC 端 |
| 信令/同步 | Node.js + Socket.io | 房间管理、状态广播、权限指令、WebRTC 信令中转 |
| 语音 | 原生 WebRTC（P2P Mesh） | 端到端音频，无媒体服务器 |
| NAT | 公共 STUN + 可选 coturn | ICE 候选/中继 |
| 网关 | Nginx | 反向代理与 TLS 终止 |
| 部署 | Docker Compose | 服务编排 |

**关键流程**：加入（房间号+密码/token 校验）→ Socket.io 建立信令连接 →（经信令交换 SDP/ICE）建立 P2P 音频 Mesh → 拉取 `latestState` 快照即时同步 → iframe 渲染演示内容并跟随。

### 7.4 语音层可替换设计（前瞻）
把语音抽象为 `VoiceProvider` 接口（`join / leave / mute / onRemoteStream`）。MVP 实现 `MeshProvider`；未来需要大场景（>10 人）时新增 `LiveKitProvider`，应用层代码不改动。**MVP 不实现 LiveKit，仅预留接口。**

## 8. 关键指标（成功标准）

- 从打开链接到画面同步 < 3s。
- 演示同步操作可感知延迟 < 150ms。
- 主持人重连后 5s 内恢复房间状态。
- 部署：一条命令，1C2G 可跑。

## 9. 里程碑

1. **M1**：建房/加入 + 上传 HTML 同源托管 + iframe 渲染 + 分享链接。
2. **M2**：同步引擎（视口锚点滚动 + 选择器点击 + 页内导航）——最难、最核心。
3. **M3**：跟随/脱离/回归 + 主持人权限（静音/拉回/**移交**）+ 断线重连（`latestState` 快照）。
4. **M4**：P2P Mesh 语音（≤10 自由发言 / 10~20 PTT）+ 静音状态 + Docker Compose（含可选 coturn）+ 文档。

## 10. 风险与对策

| 风险 | 对策 |
|------|------|
| 上传 HTML 含恶意脚本 | sandbox iframe、CSP、外链/表单白名单、体积与类型校验 |
| 跨设备 DOM 结构差异导致同步错位 | 视口锚点 + 稳定选择器；避免绝对坐标 |
| 对称 NAT 下 P2P 建连失败 | 可选 coturn 中继兜底 |
| Mesh 人数增长后连接/编码开销爆炸 | 10 人以上强制 PTT 压带宽；硬上限约 20 人；>20 走 v1.1 LiveKit |
| 弱网下同步事件堆积 | 事件去抖 + 断连仅补最新状态 |

## 11. 已定决策与待确认

**已定（本轮确认）**
- 分享链接 token：默认 24 小时有效、可配置、随房间销毁失效、非一次性。
- 语音：≤10 人自由发言，10~20 人强制 PTT，**Mesh 硬上限 20 人（MVP 冻结值）**；真·不限人数走 v1.1 LiveKit。
- 主持权移交：纳入 MVP（M3）。

**待确认**
- 真·不限人数（LiveKit v1.1）的优先级与触发时机？
